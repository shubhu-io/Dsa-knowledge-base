# Docker Tutorial

## Introduction to Docker

Docker is a platform for developing, shipping, and running applications in containers. Containers package an application with all its dependencies into a standardized unit for software development.

### Why Use Docker?
- **Consistency**: "Works on my machine" problems disappear
- **Isolation**: Applications and dependencies don't interfere with each other
- **Efficiency**: Share OS kernel, lightweight compared to VMs
- **Scalability**: Easy to scale applications horizontally
- **Portability**: Run anywhere Docker runs

### Core Concepts

#### Images vs Containers
- **Image**: Read-only template with instructions for creating a container
- **Container**: Runnable instance of an image

#### Dockerfile
A text file containing instructions to build a Docker image.

```dockerfile
# Use an official Python runtime as a parent image
FROM python:3.8-slim

# Set the working directory to /app
WORKDIR /app

# Copy the current directory contents into the container at /app
COPY . /app

# Install any needed packages specified in requirements.txt
RUN pip install --no-cache-dir -r requirements.txt

# Make port 80 available to the world outside this container
EXPOSE 80

# Define environment variable
ENV NAME World

# Run app.py when the container launches
CMD ["python", "app.py"]
```

#### Docker Compose
Tool for defining and running multi-container Docker applications.

```yaml
version: '3'
services:
  web:
    build: .
    ports:
     - "5000:5000"
  redis:
    image: "redis:alpine"
```

## Working with Containers

### Basic Container Commands
```bash
# Run a container from an image
docker run hello-world

# Run a container interactively
docker run -it ubuntu bash

# Run a container in detached mode
docker run -d -p 80:80 nginx

# List running containers
docker ps

# List all containers (including stopped)
docker ps -a

# Stop a container
docker stop <container_id>

# Remove a container
docker rm <container_id>

# Remove all stopped containers
docker container prune
```

### Container Lifecycle
```
create -> start -> running -> stop -> deleted
                    ^                    |
                    |__________ restart _|
```

## Docker Networking

### Network Drivers
1. **bridge**: Default network driver (isolated network)
2. **host**: Removes network isolation between container and host
3. **overlay**: Connects multiple Docker daemons together
4. **macvlan**: Assign MAC address to container, appearing as physical device
5. **none**: Disable networking

### Creating and Using Networks
```bash
# Create a bridge network
docker network create --driver bridge my-network

# Run a container on a specific network
docker run --network=my-network -d nginx

# Connect running container to network
docker network connect my-network <container_id>

# Disconnect container from network
docker network disconnect my-network <container_id>
```

### Network Inspection
```bash
# List networks
docker network ls

# Inspect network details
docker network inspect my-network
```

## Data Persistence

### Volumes
Preferred mechanism for persisting data generated by and used by Docker containers.

```bash
# Create a volume
docker volume create my-volume

# Start container with volume mounted
docker run -d \
  --name devtest \
  -v my-volume:/app \
  nginx:latest

# List volumes
docker volume ls

# Inspect volume
docker volume inspect my-volume

# Remove volume
docker volume rm my-volume
```

### Bind Mounts
Mount a file or directory from the host machine into a container.

```bash
# Mount host directory into container
docker run -d \
  -it \
  --name devtest \
  -v "$(pwd)"/src:/usr/src/app \
  python:3.8-slim \
  bash
```

### tmpfs Mounts
Store data in the host's memory only, never written to the filesystem.

```bash
docker run -d \
  -it \
  --name tmptest \
  --tmpfs /app \
  alpine:latest \
  ls -la /
```

## Docker Images

### Image Layers
Each instruction in a Dockerfile creates a layer in the image.
Layers are cached and reused for faster builds.

### Building Images
```bash
# Build image from Dockerfile in current directory
docker build -t my-image:tag .

# Build with no cache
docker build --no-cache -t my-image:tag .

# Build with build arguments
docker build --build-arg HTTP_PROXY=http://10.20.30.2:1234 -t my-image:tag .
```

### Image Management
```bash
# List images
docker images

# Remove image
docker rmi my-image:tag

# Prune unused images
docker image prune

# Save image to tar file
docker save -o my-image.tar my-image:tag

# Load image from tar file
docker load -i my-image.tar
```

### Image Registries
- Docker Hub (default registry)
- Private registries (self-hosted or cloud provider)
- Google Container Registry
- Amazon Elastic Container Registry
- Azure Container Registry

```bash
# Login to registry
docker login

# Push image to registry
docker push username/repository:tag

# Pull image from registry
docker pull username/repository:tag
```

## Docker Compose in Depth

### Compose File Structure
```yaml
version: '3.8'  # Specify Compose file version

services:
  web:
    build: .                    # Build from Dockerfile in current dir
    ports:
      - "5000:5000"            # Host:Container port mapping
    volumes:
      - ./code:/code           # Bind mount
      - logvolume01:/var/log   # Named volume
    environment:
      - DEBUG=1                # Environment variables
    depends_on:                # Service dependencies
      - db

  db:
    image: postgres:13
    volumes:
      - db-data:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: example

volumes:
  logvolume01:
  db-data:
```

### Compose Commands
```bash
# Start all services
docker-compose up

# Start in detached mode
docker-compose up -d

# Stop and remove containers, networks, images, volumes
docker-compose down

# Start specific service
docker-compose up web

# View logs
docker-compose logs -f

# Scale service
docker-compose up --scale worker=3

# Execute command in running container
docker-compose exec web python manage.py migrate
```

## Docker Swarm (Orchestration)

### Swarm Concepts
- **Node**: Individual Docker engine participating in swarm
- **Manager Node**: Responsible for cluster management
- **Worker Node**: Receives and executes tasks from managers
- **Service**: Definition of tasks to execute on nodes
- **Task**: Atomic scheduling unit of swarm

### Creating a Swarm
```bash
# Initialize a swarm (becomes manager)
docker swarm init --advertise-addr <MANAGER-IP>

# Join worker nodes to swarm
docker swarm join --token <TOKEN> <MANAGER-IP>:2377

# Join additional managers
docker swarm join-token manager
```

### Deploying Services
```bash
# Create service
docker service create \
  --name web \
  --replicas 3 \
  --publish 80:80 \
  nginx:latest

# List services
docker service ls

# Scale service
docker service scale web=5

# Update service
docker service update --image nginx:alpine web

# Remove service
docker service rm web
```

### Stacks with Compose
```bash
# Deploy stack from compose file
docker stack deploy -c docker-compose.yml mystack

# List stacks
docker stack ls

# List services in stack
docker stack services mystack

# Remove stack
docker stack rm mystack
```

## Docker Security Best Practices

### Image Security
- Use official base images when possible
- Scan images for vulnerabilities (docker scan, Trivy, Clair)
- Sign images with Docker Content Trust
- Minimize base images (use Alpine, distroless)
- Don't run as root inside containers

### Runtime Security
- Drop unnecessary capabilities
- Use read-only filesystems when possible
- Limit resources (memory, CPU)
- Use user namespaces
- Enable content trust
- Use seccomp, AppArmor, or SELinux profiles

### Example Secure Dockerfile
```dockerfile
# Use minimal base image
FROM alpine:3.14

# Create non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Install dependencies
RUN apk add --no-cache \
    python3 \
    py3-pip

# Copy application code
COPY --chown=appuser:appgroup . /app
WORKDIR /app

# Install Python dependencies
RUN pip install --no-cache-dir -r requirements.txt

# Switch to non-root user
USER appuser

# Expose port
EXPOSE 8080

# Run application
CMD ["python", "app.py"]
```

#### Runtime Security Flags
```bash
docker run -d \
  --name secure-app \
  --read-only \
  --tmpfs /tmp \
  --tmpfs /var/log \
  --security-opt no-new-privileges:true \
  --cap-drop ALL \
  --pids-limit 100 \
  --memory 512m \
  --cpus 0.5 \
  my-image
```

## Monitoring and Debugging

### Container Inspection
```bash
# Get detailed container info
docker inspect <container_id|name>

# Check container processes
docker top <container_id|name>

# View container logs
docker logs -f <container_id|name>

# Stats on resource usage
docker stats <container_id|name>
```

### Debugging Techniques
```bash
# Execute command in running container
docker exec -it <container_id|name> sh

# Copy files from container
docker cp <container_id|name>:/path/to/file ./local-file

# Copy files to container
docker cp ./local-file <container_id|name>:/path/to/file

# Attach to running container
docker attach <container_id|name>

# Create debug container sharing namespaces
docker run -it --pid:container:<target> --net:container:<target> \
  nicolaka/netshoot
```

### Health Checks
```dockerfile
HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:8080/health || exit 1
```

## Continuous Integration/Deployment

### CI/CD Pipeline Integration
```bash
# Build and test in CI
docker build -t my-app:$CI_COMMIT_SHA .
docker run --rm my-app:$CI_COMMIT_SHA npm test

# Push to registry after successful tests
docker push my-registry/my-app:$CI_COMMIT_SHA

# Deploy using orchestration platform
# (Kubernetes example)
kubectl set image deployment/my-app my-app=my-registry/my-app:$CI_COMMIT_SHA
```

### Blue-Green Deployment with Docker
```bash
# Deploy new version as green service
docker service create \
  --name my-app-green \
  --image my-registry/my-app:$NEW_VERSION \
  --publish 8080:80 \
  --label com.example.deployment=green

# Switch traffic to green (using load balancer)
# Scale blue to 0, green to desired count
docker service scale my-app-blue=0
docker service scale my-app-green=5
```

## Performance Optimization

### Build Optimization
- Use multi-stage builds to reduce final image size
- Leverage build cache effectively
- Combine RUN instructions to reduce layers
- Use .dockerignore to exclude unnecessary files

### Multi-Stage Build Example
```dockerfile
# Builder stage
FROM golang:1.16-alpine AS builder
WORKDIR /app
COPY . .
RUN go build -o main .

# Production stage
FROM alpine:3.14
WORKDIR /root/
COPY --from=builder /app/main .
CMD ["./main"]
```

### Runtime Performance
- Allocate appropriate resources (--memory, --cpus)
- Use appropriate storage driver for workload
- Minimize layers in images
- Use minimal base images
- Apply process init systems (tini, dumb-init) for proper signal handling

## Common Use Cases

### Microservices Architecture
```yaml
# docker-compose.yml for microservices
version: '3.8'
services:
  user-service:
    build: ./user-service
    ports: ["8001:8001"]
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/users
    depends_on:
      - db

  order-service:
    build: ./order-service
    ports: ["8002:8002"]
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/orders
    depends_on:
      - db

  db:
    image: postgres:13
    volumes:
      - postgres-data:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: secret

volumes:
  postgres-data:
```

### Development Environments
```yaml
# docker-compose.yml for development
version: '3.8'
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile.dev
    volumes:
      - ./src:/app/src      # Live code reload
      - ~/.ssh:/root/.ssh   # SSH keys for git
      - ~/.aws:/root/.aws   # AWS credentials
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
    command: npm run dev

  db:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: example
      MYSQL_DATABASE: myapp
    volumes:
      - db-data:/var/lib/mysql

volumes:
  db-data:
```

### Testing Environments
```bash
# Run tests in isolated container
docker run --rm \
  -v "$(pwd)":/app \
  -w /app \
  node:16 \
  npm test

# Parallel testing with compose
docker-compose -f docker-compose.test.yml up --abort-on-container-exit
docker-compose -f docker-compose.test.yml down -v
```

## Troubleshooting Common Issues

### "Cannot connect to the Docker daemon"
```bash
# Start Docker service
sudo systemctl start docker

# Or start daemon manually
sudo dockerd

# Check if user is in docker group
groups $USER
# Add user to docker group if needed
sudo usermod -aG docker $USER
newgrp docker
```

### "No space left on device"
```bash
# Check disk usage
docker system df

# Prune unused data
docker system prune -a

# Clean volumes
docker volume prune

# Clean builder cache
docker builder prune
```

### "Conflict. The container name is already in use"
```bash
# Remove stopped container with same name
docker rm existing-container

# Or rename existing container
docker rename existing-container old-name

# Or use different name for new container
```

### "Permission denied" while accessing volume
```bash
# Check ownership of mounted directory
ls -ld /host/path

# Fix ownership if needed
sudo chown -R $(id -u):$(id -g) /host/path

# Or run container with matching UID/GID
docker run -u $(id -u):$(id -g) -v /host/path:/container/path image
```

## Docker Alternatives and Complementary Technologies

### Container Runtime Alternatives
- **containerd**: Industry-standard container runtime (used by Docker and Kubernetes)
- **cri-o**: Lightweight runtime for Kubernetes
- **runc**: Low-level container runtime
- **podman**: Daemonless container engine (compatible with Docker CLI)

### Orchestration Platforms
- **Kubernetes**: Most popular container orchestrator
- **Apache Mesos**: Generalized cluster manager
- **Nomad**: Flexible workload orchestrator by HashiCorp
- **OpenShift**: Kubernetes distribution by Red Hat

### Service Meshes (often used with containers)
- **Istio**: Traffic management, security, observability
- **Linkerd**: Lightweight service mesh
- **Consul Connect**: Service mesh from HashiCorp

### Monitoring and Logging Solutions
- **Prometheus + Grafana**: Metrics collection and visualization
- **ELK Stack** (Elasticsearch, Logstash, Kibana): Log aggregation
- **Fluentd**: Unified logging layer
- **Jaeger**: Distributed tracing

## Summary

Docker revolutionized application deployment by introducing lightweight, portable containers. Key takeaways:

1. **Images are immutable templates**, containers are mutable runtime instances
2. **Layered filesystem** enables efficient storage and sharing
3. **Volumes** provide persistent storage separate from container lifecycle
4. **Networking** offers multiple models for different isolation needs
5. **Compose** simplifies multi-container application definition
6. **Swarm** provides native clustering and orchestration
7. **Security** requires attention at build time and runtime
8. **Best practices** include minimal images, non-root users, and resource limits

Docker continues to evolve with features like BuildKit for improved builds, rootless mode for enhanced security, and better integration with orchestration platforms like Kubernetes. Understanding these concepts enables developers to build, deploy, and manage applications more efficiently and reliably.
