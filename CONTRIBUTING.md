# Contributing Guidelines

Thank you for considering contributing to this Computer Science & Software Engineering Knowledge Base! We welcome contributions from everyone.

## How to Contribute

There are many ways to contribute:

1. **Report bugs** or suggest improvements by opening an issue
2. **Improve existing content** by fixing errors, enhancing explanations, or updating outdated information
3. **Add new content** on topics not yet covered
4. **Add practice problems** with solutions
5. **Improve code examples** in various languages
6. **Add visual aids** like diagrams or code annotations
7. **Help with translations** to make content accessible globally
8. **Review pull requests** from other contributors

## Getting Started

1. Fork the repository
2. Clone your fork locally:
   ```bash
   git clone https://github.com/your-username/complete-cs-knowledge-base.git
   cd complete-cs-knowledge-base
   ```
3. Create a new branch for your contribution:
   ```bash
   git checkout -b feature/your- feature-or-fix-name
   ```
4. Make your changes
5. Commit your changes with a descriptive message
6. Push to your fork
7. Submit a pull request

## Content Guidelines

### Topic Structure
Each topic should ideally include:
- Overview/Introduction
- Theory with clear explanations
- Visual explanations/diagrams (ASCII art or descriptions)
- Implementation in multiple languages (C/C++, Java, Python, JavaScript)
- Time and Space complexity analysis
- Real-world applications
- Common interview questions
- Practice problems with solutions
- Common pitfalls and how to avoid them
- Further resources (books, videos, articles)

### File Naming Convention
- Use lowercase with hyphens as separators: `topic-name.md`
- For subtopics: `topic-name/subtopic-name.md`
- Code files should follow language conventions:
  - C/C++: `.c`, `.cpp`
  - Java: `.java`
  - Python: `.py`
  - JavaScript: `.js`

### Content Quality
- **Accuracy**: Ensure technical correctness
- **Clarity**: Explain concepts in simple terms before diving deep
- **Completeness**: Cover beginner to advanced aspects
- **Practicality**: Focus on what's useful in real interviews and jobs
- **Engagement**: Use examples, analogies, and interactive elements where possible

### Code Standards
- Write clean, readable, and well-commented code
- Follow language-specific best practices and conventions
- Include time/space complexity comments
- Provide multiple approaches when relevant (brute force → optimized)
- Test code examples when possible

### Language Consistency
When providing code examples in multiple languages, maintain this order:
1. C
2. C++
3. Java
4. Python
5. JavaScript

### Markdown Formatting
- Use proper heading hierarchy (# for main title, ## for sections, etc.)
- Use code fences with language specifiers for syntax highlighting
- Use tables for comparisons
- Use blockquotes for important notes or warnings
- Keep line length reasonable (< 100 characters)
- Use consistent terminology

## Specific Contribution Types

### Adding a New Topic
1. Create the appropriate directory if it doesn't exist
2. Create a main `topic-overview.md` file
3. Add subtopic files as needed
4. Update any relevant README or navigation files
5. Add practice problems in the appropriate practice section

### Improving Existing Content
- Fix factual errors
- Improve explanations that are unclear
- Add missing programming language implementations
- Include additional edge cases in explanations
- Enhance visual explanations
- Update outdated information

### Adding Practice Problems
- Clearly state the problem
- Provide input/output format
- List constraints
- Give examples with explanations
- Provide hints (without giving away solution)
- Include solution with explanation
- Mention time/space complexity
- List related problems

### Adding Code Examples
- Make sure code compiles and runs
- Include necessary imports/includes
- Use meaningful variable names
- Add comments explaining key logic
- Handle edge cases
- Follow language-specific style guides

## Review Process

1. All contributions should be made as pull requests
2. PRs should target the `main` branch
3. Include a clear description of what was changed and why
4. Reference any related issues
5. At least one maintainer will review the PR
6. Be open to feedback and willing to make changes
7. Once approved, maintainers will merge the PR

## Code of Conduct

Please note that this project is released with a Contributor Code of Conduct. By participating in this project you agree to abide by its terms.

## Questions?

If you have questions about contributing, please open an issue or reach out to the maintainers.

Thank you for helping make this resource better for everyone!
